# Change Decisions

This file records user decisions that change how Thoth should be interpreted.

## Decisions

1. `NTH-CD-001` `[2026-06-27]`: The archived plugin form is the final plugin archive. It is no longer maintained. Archive release: `thoth-plugin-final-archive`. Archive branch: `archive/main-20260627`.
2. `NTH-CD-002` `[2026-06-28]`: Thoth active implementation starts from the TypeScript line. Archived plugin code should be removed from the active working tree after enough design information is extracted.
3. `NTH-CD-003` `[2026-06-28]`: The Thoth design authority is the core principles document plus three canonical design documents: high-level design, MVP user journey and engineering architecture.
4. `NTH-CD-004` `[2026-06-28]`: The hardest product problems are clarify quality and aggressive loop design, not ordinary software development.
5. `NTH-CD-005` `[2026-06-28]`: Thoth should behave like a real private secretary for a high-authority user. It should remember context and ask only a few golden questions when human judgment is truly needed.
6. `NTH-CD-006` `[2026-06-28]`: A task loop must aggressively solve the previous unresolved problem. It must not mechanically repeat runs or only add incremental tests.
7. `NTH-CD-007` `[2026-06-28]`: The active repo reset used `GPL-3.0-only`, `npm workspaces`, package namespace `@thoth/*`, version `0.0.0`, and package skeletons marked `private: true`. The license part is superseded by `NTH-CD-017`.
8. `NTH-CD-008` `[2026-06-28]`: TUI must use OpenTUI. Node/Bun runtime choice is intentionally deferred to the first TUI spike.
9. `NTH-CD-009` `[2026-06-28]`: Archived prompt assets are preserved as docs-only structured contract seeds, not as executable Python code.
10. `NTH-CD-010` `[2026-06-28]`: `CLAUDE.md` should be a symlink to `AGENTS.md` in this reset.
11. `NTH-CD-011` `[2026-06-29]`: Thoth should later adopt a Paseo-like release and packaging pipeline. The intended shape is tag-triggered GitHub Actions rather than ordinary branch-push publishing: desktop installers are built with Electron Builder for macOS, Linux and Windows and uploaded to GitHub Releases; Android APK release builds use Expo/EAS and are uploaded to GitHub Releases; web/app deployment can use Cloudflare Pages or an equivalent hosted target; iOS should not be treated as a GitHub Release IPA self-install path and needs a separate TestFlight/App Store/EAS submit distribution decision.
12. `NTH-CD-012` `[2026-06-29]`: Thoth itself is not a harness tool and must not become a hidden LLM API wrapper. All AI and agent execution capability comes from configured providers: ACP adapters, harness runtimes, app-server sessions, official harness SDK/control surfaces or local harness CLIs. Thoth source owns process flow, routing judgment, preset prompt contracts, task authority, frozen acceptance, evidence and session records; it must not privately call general model inference APIs as a substitute for provider sessions.
13. `NTH-CD-013` `[2026-06-29]`: Thoth must not use local deterministic heuristics for zero-shot semantic routing. Any intelligent judgment about intent type, workspace context, ambiguity, clarification questions, route upgrade/downgrade or execution strategy must run inside a provider-backed session. The desktop composer should expose explicit Thoth-level controls for task mode (`quick`, `loop`), clarification strength and loop strength; local code may honor those explicit controls and run mechanical authority checks, but it must not silently infer task type from natural language.
14. `NTH-CD-014` `[2026-06-29]`: The chatbox composer controls are locked as `+`, Provider, Mode, Clarify and Loop. `+` only supports image attach and file upload under `10MB` for MVP. Scope is not a separate button; users reference scope with `@`. Provider contains provider/model/runtime settings including model id, thinking strength, permission mode and fast mode. Mode has two options: `Quick` and `Loop`. Clarify has five user-facing levels: `auto`, `no-ask`, `light`, `balance`, `deep`, and applies to both `Quick` and `Loop`; protocol enums may normalize them as `auto`, `no_ask`, `light`, `balanced`, `deep`. Loop has five user-facing levels: `auto`, `no-loop`, `light`, `balanced`, `endless`; it only applies when Mode is `Loop`, otherwise it is disabled/greyed out. `endless` is red, high-cost, and runs until the user manually stops it, while still respecting safety, permission, provider availability and resource boundaries.
15. `NTH-CD-015` `[2026-06-29]`: The business flow is updated around provider-native sessions. All visible provider output must stream through Thoth in real time. `Quick + Don't Bother Me` is a Paseo-like provider passthrough path. `Loop` uses a read-only Clarify provider session, then a frozen contract, then one PlanExec provider session using provider-native plan mode when available, then an independent Review provider session. Clarify may read workspace state and search the web but must not mutate files. After contract freeze, PlanExec provider clarification questions are auto-answered from the frozen contract or first recommended option and recorded; provider permission requests are still governed by permission policy. User-facing labels supersede the old names in `NTH-CD-014`: Clarify uses `Don't Bother Me` instead of `no-ask`; Loop uses `One Plan, One Do` instead of `no-loop` and `Run Until Stopped` instead of `endless`.
16. `NTH-CD-016` `[2026-06-30]`: Clarify is locked as a decision-tree walk rather than a questionnaire. It must ask the current highest-leverage root or branch decision first, after resolving discoverable facts through read-only investigation. Clarify provider output is split into visible stream text and structured interaction candidates. Visible provider text streams to clients in real time, but clarification cards render only after Thoth validates the structured candidate. Invalid card candidates, schema errors and repair prompts are hidden from the user; the daemon sends a concise repair report back into the same provider session and asks it to regenerate the same card for the same tree node. Client UI keeps a lightweight defensive validator, while the daemon owns canonical validation and repair authority.
17. `NTH-CD-017` `[2026-06-30]`: Thoth switches to `AGPL-3.0-or-later`. Implementation seed import uses only the latest GitHub upstream source from `getpaseo/paseo`; the local dirty checkout is ignored. Raw upstream material lives only in ignored `.agent-os/upstreams/paseo/`; tracked seed material may live in `_paseo/` directories and may temporarily be broken. Multica source code must not be copied. Voice, audio, speech and dictation material is excluded from raw cache and tracked seeds.

Voice/audio/speech/dictation product capability remains out of scope. Broken imports, scripts, dependencies and type errors are expected until dependency and compile triage.

31. `NTH-CD-031` `[2026-07-03]`: Development of the `thoth.clarify` and `thoth.loop` runtime skills must be golden-driven and independently judged. Each skill must have fixed golden data that captures expected agent behavior, not only schema shape. During development, Codex must run current skill/prompt/rubric behavior against those golden cases and use an independent session, normally through `codex exec`, to judge whether the generated questions, execution behavior, review behavior and retry guidance satisfy behavioral-psychology principles, behavior-tree convergence, frozen goals/constraints/acceptance and low user cognitive load. The main development session must not self-certify skill quality from packet validity or passing unit tests alone. Judge failures such as asking irrelevant questions, pushing agent-discoverable facts back to the user, failing to converge, repeating low-value questions, ignoring constraints, wasting user attention or mechanically retrying the same failed strategy require changing the skill, prompt or rubric and re-running the golden review.
32. `NTH-CD-032` `[2026-07-03]`: The final two Clarify confirmation packets must preserve provenance mechanically. When `thoth.clarify` reaches `C_TASK_CARD`, Thoth must attach the full prior Clarify question/answer transcript verbatim inside packet `content`; the Task Card is grounded in that transcript rather than implicit provider memory. When `thoth.clarify` reaches `C_GOAL_CARD`, Thoth must attach both the full prior Clarify transcript verbatim and the exact first-round CEO Task Card that the user approved, including any user-driven modifications. The Goal Card split must be grounded in the approved CEO Task Card plus transcript; it must not invent, drop, reinterpret or over-expand goals, constraints or acceptance. 33. `NTH-CD-033` `[2026-07-03]`: Loop Goal 1 and Loop Goal 2 are refined around behavior-tree branch questions and Paseo-style question-card rendering, superseding prior default-recommendation wording for Clarify. In `thoth.clarify`, a behavior tree means a decomposition tree of the user's prompt: each question should cut at the highest-leverage branch that preserves the user's stated target while excluding wrong routes, such as demo-grade versus production-grade for a Three.js PathTracing engine. Clarify must not ask fallback-scope downgrade questions that replace the user's requested target with an easier MVP, demo, mock, partial implementation, or different target. Clarify does not provide default choices by default; if a technical decision is better made by the agent, the agent should decide and record the assumption, and only provide a recommendation when the user asks for one. A greeting such as `hi` should be answered simply and warmly, like "你好 Boss，有什么需求？", without product explanation or Clarify UI. The older per-round compact-preset wording in this decision is superseded by `NTH-CD-034`: rules live in `SKILL.md`, and normal same-state packets carry runtime data rather than repeating rules. "Reference Paseo" means reusing or mirroring Paseo's existing request-user-input card rendering capability, not adopting Codex `request_user_input` or `AskUserQuestion` as the Clarify mental model. Clarify answer packets/cards should have a title and 2-4 branch choices; each choice label is at most 10 Chinese characters and each choice explanation is at most 20 Chinese characters. Users may attach a note to selected choices, or select no choice and answer with note only.
34. `NTH-CD-034` `[2026-07-04]`: Loop Goal 1 `thoth.clarify` / `thoth.loop` runtime skill handling is standardized. `thoth.clarify` and `thoth.loop` must be standard Skill artifacts with `SKILL.md` as the canonical source, created and maintained through the standard skill-create / skill creator workflow, with YAML frontmatter, `name`, `description`, Markdown body and optional `references/`, `scripts/` or `assets/`. They must not use Codex-only fields, Codex-only hooks, Codex-only metadata or Codex-only trigger formats, and must remain loadable or equivalently invocable by Claude Code, Codex, ACP-compatible providers and future harness runtimes. Thoth-specific metadata such as `x-thoth-runtime`, `x-thoth-required` and `x-thoth-scope` is allowed only as Thoth internal interpretation, not as a provider-compatibility precondition. Skill source is visible, maintainable and reviewable, but `user-invocable: false` means ordinary users cannot manually select, call, disable or see raw runtime packets for the skill. Thoth must not install these internal runtime skills into user global provider skill dirs such as `~/.codex/skills`, `~/.claude/skills` or `~/.agents/skills`; they live in Thoth-owned internal runtime skill bundles and are mounted only inside Thoth-owned provider session scope. If a provider supports native session skills, Thoth uses that; if it only supports filesystem discovery, Thoth creates an isolated provider/session skill home; if no scoped skill support exists, hidden prompt rendering is a compatibility fallback rather than the primary path. `SKILL.md` holds mental model, runtime arguments, state codes, transition rules, question rules, good/bad cases and output contract; TypeScript may load, validate, hash, mount, invoke or fallback-render it, but TS prompt constants are not the canonical behavior source. Normal same-state packets must not repeat Skill rules or include `skill_ref`; they carry runtime data such as current state, user input, transcript ref/delta, context summary and provenance refs. Session start, state transition, skill digest/version change, context loss and repeated repair failure may carry `skill_ref` / digest / `according_to_loaded_skill`, but must not copy the full rules. Provider semantic transition reasoning belongs in `SKILL.md`; daemon/protocol own only a mechanical transition table that checks allowed transitions, packet completeness and provenance presence. Repair packets repair shape/state/provenance only, do not reinterpret user intent, do not create new questions unless the intended state was already `C_ASK`, do not change transcript, target or approved CEO Task Card, and keep bad packets/repair logs out of user-visible transcript. Loop-1 acceptance now requires an independent `codex exec` user-interaction simulation based on the installed runtime artifact, not only golden fixture review; that simulation must also prove the internal runtime skill mount does not pollute bare provider skill homes.
35. `NTH-CD-035` `[2026-07-04]`: Loop Goal 1 Clarify strength is now an explicit behavior strategy rather than a passive field. `none` means direct answer/execute with no proactive Clarify except safety, permission, irreversible, hard contradiction or impossible-work blockers; `light` asks only the highest-impact behavior-tree fork; `balanced` asks that core fork plus 1-2 material leaves around goal, constraints, acceptance, risk or resources; `dive` walks all material assumptions while still rejecting implementation minutiae, common-sense questions, standard answers and agent-discoverable facts. Legacy `deep` may remain as a compatibility alias, but the canonical new wording is `dive`. `thoth.clarify` must classify assumptions as `user_must_decide`, `agent_can_decide`, `agent_can_discover` or `standard_answer/common_sense`; only high-impact `user_must_decide` assumptions become visible questions. Normal same-state packets now carry compact runtime controls with `clarify_strength` / `effective_clarify_strength`, plus transcript, assumption ledger and decision-tree frontier refs when available, but still do not carry `skill_ref` or repeat Skill rules. Transition packets may carry `controls_changed` when strength changes. `C_ASK` question cards now represent one titled card with 2-4 tightly related questions, each with short choices and optional notes, rather than only one question with 2-4 choices; `content.meta` records hidden internal data such as effective strength, decision-tree depth, QA round count, remaining material assumptions and question-value reason. These meta fields, state codes, repair details, skill ids and raw packets are not user-visible. Golden/eval/judge acceptance must prove the same prompt behaves differently under `none` / `light` / `balanced` / `dive`, that agent-discoverable facts are not pushed to the user, and that user stop signals like "够了/不要再问" converge to a grounded Task Card when transcript is sufficient.
