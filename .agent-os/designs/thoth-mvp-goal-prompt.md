`thoth.clarify` 和 `thoth.loop` 的开发必须 golden-driven。不要只证明 packet/schema 合法；必须准备固定 golden 数据，记录输入、上下文、期望行为树节点、目标/约束/验收、禁止问题或禁止策略、低用户心智负担标准。每次修改 skill、prompt、rubric 或关键行为约束后，先跑 golden，再用独立 session，通常通过 `codex exec`，审查当前 transcripts 是否符合行为心理学、行为树收敛、目标/约束/验收和低 cognitive load。主开发 session 不能自评通过；如果 judge 发现无关追问、字段问卷化、把可发现事实推给用户、没有收敛、忽略 frozen contract、Review 只等同跑测试或 retry 机械重复，必须改 skill/prompt/rubric 后重跑。

Loop 1/2 的 Clarify 最新口径：行为树是用户 prompt 的拆解树，每轮只问当前最高价值分叉，保留用户原始目标并快速排除错误路线。禁止兜底降级问题：不得把用户要求的 A 降级成更容易的 MVP、demo、mock、局部实现或 B。默认不提供推荐或默认选项；技术判断 agent 更适合做时就自己判断并记录，只有用户要求推荐时才推荐。`clarify_strength` 必须影响行为：`none` 表示不进入 Clarify、直接裸 provider foreground stream；`light` 只问最高影响核心分叉；`balanced` 深入 1-2 层 material leaves；`dive` 尽量消除 material assumptions 但不问实现细枝末节、常识、standard answer 或可发现事实。`thoth.clarify` 必须区分 `user_must_decide`、`agent_can_decide`、`agent_can_discover`、`standard_answer/common_sense`，只把高影响 `user_must_decide` 问给用户。`thoth.clarify` 的行为规则写在标准 `SKILL.md` 中；普通同状态 provider input 只带 phase/state/controls、transcript ref、assumption ledger ref 和 decision-tree frontier ref，不重复 Skill 规则，不带 `skill_ref`。session start、状态转移、skill digest/version 变化、context 丢失或 repair 多次失败时才带 `skill_ref` / digest marker；强度变化可带 `controls_changed`。`hi` 在 Quick+none 下必须裸 provider stream、无 packet、无 card；在 active Clarify 下可 `C_DIRECT`，但仍要简短自然、不允许 daemon 本地固定问候。

Clarify question/answer card 可参考 Paseo 现有 Codex request-user-input card 的渲染能力，但产品语义仍是隐藏 `thoth.clarify`。每个 question card 有标题、一个 primary 行为树节点和 2-4 道紧密相关的问题；每道题有短分支选择，每个选择 label 不超过 10 个中文字，解释不超过 20 个中文字。Answer packet 支持给每道题被选选择加 note，也支持不选任何项只写 note。`C_ASK` 可以带 hidden `content.meta` 记录 effective strength、tree depth、QA round count、remaining material assumptions 和 question value reason，但这些不直接展示给用户。

`thoth.clarify` 最后两轮确认必须保留 provenance。`C_TASK_CARD` packet 的 `content` 必须携带此前所有 Clarify 问答原文 transcript；`C_GOAL_CARD` packet 的 `content` 必须同时携带该 transcript 和第一轮用户确认过的总 CEO Task Card 原文。Goal 分拆必须 grounded in transcript + approved CEO card，不能只靠 provider 隐式记忆，不能臆造、丢失、篡改或过度解释目标、约束、验收和风险。

必须优先实现的 MVP 链路：原版 Paseo composer 输入用户 prompt，附带 Provider / Clarify / Mode 控件、workspace/session/topic context。Mode=Quick 且 Clarify=none 时走裸 provider stream，不进入 Clarify packet。Mode=Quick 且 Clarify 非 none 时，daemon 构造 phase-aware provider input envelope，provider session 通过 `thoth.clarify` runtime skill 和 `submit_clarify_packet` bridge 运行；`C_ASK` 在完整 provider 输出和 daemon schema/packet 校验后 atomic 插入 Clarify card；后续 `C_TASK_CARD` / `C_GOAL_CARD` 同样完整校验后 atomic 渲染；用户确认后进入同 session `quick_exec` 并普通流式执行。Mode=Loop 时，确认后才 `C_REGISTER`，后台由独立 PlanExec/Review sessions 执行。packet 不合格时进入 repair/blocker；高风险动作必须通过 permission card。

Loop-2 前端特别约束：这是硬 APP 前端重构 gate，但方向不是继续 toy Workspace Secretary 三视图，而是恢复/保持原版 Paseo frontend app 能力和布局。`packages/app/src/thoth-app/thoth-app-shell.tsx` 这类 toy shell 不能作为主入口；能删就删，短期有类型依赖时只能隔离成 legacy/deprecated 非主路径。清理主界面的 toy/debug/验收/合同语言：`WORKSPACE SECRETARY`、`当前需求收敛`、`Quick 前台 · Loop 后台`、`真实 provider 已连接`、`Quick 和 Loop 都会通过真实 provider 结果写入历史`、`当前秘书话题`、`新秘书话题`、完整本地路径、`provider-backed clean UI model`、`C_DIRECT` / `C_ASK` / packet / repair / schema / raw JSON / provider role 等。Clarify card 必须是 Thoth decision card，支持 2-4 道紧密相关问题、短选项、短解释、per-option note、note-only、你推荐、你决定，没有默认预选或默认推荐，并稳定渲染在 Paseo transcript / agent-stream 中。UI 只消费 typed clean UI model / clean streaming event，不能解析 assistant 文本、markdown JSON、code fence、snippet 或 raw packet，不能本地判断收敛、生成 Task/Goal Card、修改 authority 或替用户选择。
