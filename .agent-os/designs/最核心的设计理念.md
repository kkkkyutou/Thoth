# 最核心的设计理念

## Status

1. 日期：`2026-06-28`
2. 性质：New Thoth 最核心 high-level 目标摘要
3. 范围：只记录最重要的产品设计理念，不展开工程架构、代码结构或实现路径

## 一句话

Thoth 最重要的设计理念是：以真实私人秘书的方式理解用户，把用户模糊的意图变成可验证、可恢复、可异步执行、可审查的任务 loop，并且尽可能降低使用门槛，让用户只做真正需要人判断的决策。

## 1. 最大程度减少用户心智负担和使用门槛

这是 Thoth 的第一目标，也是所有设计取舍的最高裁判。

用户不应该管理 provider、模型、session、权限细节、上下文、日志、重试和验收路径。用户也不应该学习一套新的复杂操作方法，才能让 Thoth 正确工作。用户只应该管理目标、边界、风险、验收和必要拍板。

用户可以在输入框旁显式选择少量 Thoth-level 控件，例如这次是回答、直接处理还是正式任务，以及澄清强度和 loop 强度。这不是把 provider 复杂度交给用户，而是让用户用很低成本表达“这件事应该被如何对待”。所有真正的语义判断仍必须通过 provider session 完成。

如果一个功能让 Thoth 更强但让用户更累，它就不是 Thoth 的核心方向。Thoth 的价值不是暴露更多 AI 能力，而是把 AI 执行过程中原本需要人持续盯着、记着、判断着的负担接过去。

这里必须按照真实人的人格来思考：假设用户是最高权限的公司 CEO，Thoth 是最重要的私人秘书。一个合格的私人秘书不会要求 CEO 每次都重新交代上下文，也不会把所有边界、细节和执行选项原样丢回给 CEO。秘书应该记得昨天 CEO 提过的项目，自然而然能对应“昨天说的那个项目怎么样了”；秘书也应该理解 CEO 的表达习惯、优先级、风险偏好和决策风格。

因此，Thoth 的交互不应该像表单，也不应该像命令行工具。Thoth 应该像一个有稳定性格、可靠记忆和主动判断力的助手：听得懂含糊表达，记得前后文，知道什么该自己查，知道什么必须问，并且只把最关键的决策交还给用户。

## 2. 把模糊意图编译成清晰任务

Thoth 最重要的能力不是执行，而是把用户一股脑说出来的想法、需求、担忧和背景，变成一个清楚的 `Loop` task。

这个 task 必须说明：

1. 目标是什么
2. 不做什么
3. 约束是什么
4. 风险在哪里
5. 验收标准是什么
6. 哪些是假设
7. 哪些必须用户拍板

因此，Thoth 的核心不是 prompt runner，而是 clarification-to-task compiler。

这里同样要按私人秘书的真实工作方式设计。用户说“帮我实现一个……”，Thoth 不应该机械地把所有边界条件、技术方案、验收细节、异常情况都丢给用户逐项填写。真正合格的秘书会先站在 CEO 的意图里深度理解：这个需求背后真正想改变什么、影响谁、最可能踩到什么风险、哪些信息可以自己调查、哪些决策必须 CEO 拍板。

然后 Thoth 只问最重要的几个黄金问题。黄金问题不是越多越好，而是每个问题都必须能显著改变执行方向、风险边界或验收标准。能由 Thoth 自己通过上下文、历史记忆、workspace 勘察、已有资料推断或验证的内容，不应该变成用户负担。不能确定且影响很大的内容，才需要问用户。

## 3. 把任务变成可恢复、可审查、可长期运行的 loop

Thoth 不应该把任务理解成一次对话或一次 agent run。

正式任务应该天然是一个 loop：可以计划、执行、失败、重试、暂停、恢复、审查和汇报。用户下发任务后可以离开，Thoth 继续推进；用户回来后能看到任务现在在哪、为什么卡住、证据是什么、下一步是否需要拍板。

loop 的关键不是“重复跑一遍”，也不是每一轮都温吞地做一点点 incremental 修补。每一轮 loop 都必须针对上一轮没有解决的问题激进推进：明确上一轮失败在哪里，为什么失败，哪个假设错了，哪个证据缺了，哪个实现路径走不通，然后集中火力解决这个核心阻塞。

如果上一轮卡在验收不充分，下一轮就应该优先补齐验收证据；如果上一轮卡在方向错误，下一轮就应该重新规划而不是继续堆小修；如果上一轮卡在实现质量，下一轮就应该主动重构关键路径而不是只补单元测试。loop 的价值在于持续逼近目标，而不是制造“系统还在忙”的假象。

所以 Thoth 的目标不是跑一次 agent，也不是机械重复测试，而是维护一个能够识别失败原因、激进修正方向、持续推进闭环的异步任务控制平面。

## 4. 让成功建立在验收和证据上

这是 Thoth 区别于普通 harness 的关键。

执行者不能自己说成功就算成功。Thoth 必须把验收标准提前冻结，并在执行后用独立 Review 检查：

1. 是否真的达成目标
2. 是否违反约束
3. 是否有证据
4. 是否漏测
5. 是否范围漂移
6. 是否只是看起来成功

Thoth 应该是 validator-first、evidence-first 的系统，而不是乐观的 agent 聊天壳。

## 5. 保持宿主无关和多端一致

未来 harness 会继续变化，Thoth 不能把核心语义绑定到任何一个宿主上。

Thoth 本身也不是 harness。它不提供自己的模型能力，不私自调用 LLM API，也不把普通模型 API 包装成“内部 agent”。所有 AI 能力、代码执行能力、工具调用能力和 agent session 都来自用户配置的 provider：ACP adapter、harness runtime、app-server、官方 harness SDK/control surface 或本地 harness CLI。Thoth 只负责流程、判断、预设 prompt、合同、验收、证据和记录。

Thoth 应该做到：

1. 用户面对的是 One Thoth
2. 内部可以调不同 harness
3. task、验收、证据、状态、记忆不属于某个 harness
4. TUI、桌面 app、手机 app 看到同一份真相
5. UI 是壳，authority 在 Thoth

这个目标保证 Thoth 不会随着某个工具兴衰而失效。
